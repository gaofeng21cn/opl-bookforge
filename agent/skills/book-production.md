# Book Production Skill

Use this skill when drafting or revising a book package from an approved storyline map.

Working policy:

- Before drafting body prose for a book-length target, inherit the target extent, allocate target pages/chars by chapter, name the active production queue, and keep that queue visible until every chapter passes its minimum.
- Treat target extent as an input to drafting prompts, not as an audit after prose exists. Each chapter prompt should include the chapter's minimum extent, current extent, missing extent, section development obligations, and case/figure/table obligations before asking for manuscript prose.
- Draft chapters from the chapter thesis chain and evidence map.
- Draft from a chapter function contract. Each chapter prompt should name the chapter's one job, its new argument movement, its handoff from and to adjacent chapters, and the adjacent claims it should avoid re-arguing.
- Keep an early concept map active during drafting. If a recurring term will do heavy work later, orient the reader before the first sustained use and deepen the definition in the assigned chapter.
- Track two or three whole-book core models through drafting. Introduce, interpret, apply, and recover those models across the manuscript so figures and tables become method-bearing structures rather than isolated summaries.
- Use a case evidence ladder before writing case claims. Constructed scenes, typical scenarios, documented process materials, authorized interviews/materials, and outcome evidence support different claim strengths.
- When owner or reviewer critique names structural quality defects, update chapter task cards, style engine, concept map, core model map, evidence ladder, or QC gates before revising prose.
- When owner or reviewer critique is small, local, and evidence-bounded, use the fast-track revision skill. Record an audit, edit the relevant Markdown/domain refs, refresh assembly metrics, hygiene, scans, and the cumulative review PDF, and do not start a new independent meta-review round unless the critique changes whole-book logic or invalidates the latest review verdict.
- Use the reader-facing draft skill during drafting and expansion. The first visible chapter body should read as book prose, not as a production brief or outline memo.
- When the owner supplies a stronger reference draft or says another version reads better, run the reference-draft-absorption skill before continuing production. Convert transferable strengths into chapter task cards, reader-entry plans, style-engine rules, QC gates, and publication-design rules; do not handle the reference as a one-off polish checklist.
- Use the reader-style contract as the live prose target for every chapter; do not write to a generic educated-reader style when the book declares a specific reader group.
- Preserve reader priority during drafting: write for the declared primary readers. Secondary readers may remain able to follow the prose, but they must not pull chapter structure, explanatory burden, examples, or tone away from the primary-reader promise unless an owner-approved contract revision says so.
- Preserve the declared author/source stance for major cases. If the author team is a designer, executor, or participant in the case, draft from that practice-involved stance and explain the authors' anticipation, design choices, implementation logic, and reflective limits. Do not write such cases as if the book were an outside observer summarizing public facts.
- Keep practice-involved cases evidence-bounded. Active "we" or author-team voice may explain design judgment and execution mechanisms, but it must not claim unprovided outcomes, interviews, authorization, user validation, learning effects, or impact metrics. Missing evidence remains a visible typed gap such as `待补真实案例`, not a reason to flatten the chapter into public-source observation.
- Before drafting visible chapter prose, add a reader-entry plan to the chapter brief or drafting notes: opening scene/question, reader tension, concrete example/case, main claim, section movement, table/figure role, and closing transition. Keep this plan out of the manuscript body and use it to draft natural prose directly.
- Keep chapter task, core question, thesis, source refs, target budget, figure asset state, blockers, and QC notes in chapter briefs, manifests, reports, comments, or handoff refs. Do not put these production metadata fields into the reader-facing manuscript body.
- Open each chapter or major section through a concrete scene, reader tension, practical question, or consequence before introducing categories or definitions.
- Keep prose in author-facing Markdown chapter files or equivalent Markdown refs. Treat scripts as thin assembly, metrics, validation, export, and report helpers; do not make generator-code string literals the manuscript source of truth.
- For book-length work, draft and review by chapter/part first, then assemble and run whole-book checks. The assembled `book.md` is a delivery/export ref, not the only creative working file unless the owner explicitly chooses that workflow.
- Preserve the target extent declared by the owner brief, source plan, publisher brief, or storyline map. Do not reduce a book-length target to a compact/sample draft unless the owner explicitly changes that target; record any shortfall as a typed extent blocker.
- Enforce chapter budgets before completion labels. A chapter below its target minimum is not drafted or done; mark it `seed_in_progress` or `draft_in_progress` and keep it in the production queue.
- Do not treat an all-chapter coverage seed as completed materialization. If every chapter exists but remains below budget, call it seed material or a preview skeleton, keep the production queue open, and complete one chapter package at a time.
- When style or cleanup edits reduce a chapter below its target minimum, repair it with substantive argument, cases, examples, figure/table interpretation, or transitions grounded in the chapter brief. Do not pad with summaries, duplicated claims, or inflated wording.
- Generate below-target whole-book assemblies only as named previews, for example `book.preview.md`; reserve final `book.md` for all-chapters-ready assemblies unless the owner explicitly approves another export meaning.
- When a compact/sample draft was produced against a book-length target, retire it as an obsolete draft and rebuild the active manuscript from chapter packages rather than expanding the invalid file in place.
- Retired drafts should become tombstone/archive records, not full searchable obsolete prose inside the active workspace. If full old text must be preserved for evidence, move it outside the active book workspace or store a non-manuscript receipt that cannot be mistaken for current source.
- Maintain a production pipeline contract for long manuscripts: chapter packages, briefs, drafts, chapter QC, figure asset manifest, table plan, whole-book review, export refs, and blockers.
- Maintain a book-memory contract for nonfiction work: working memory for the current chapter task, episodic memory for prior chapter reviews and owner decisions, and semantic memory for durable thesis, audience, source canon, glossary, style assets, and evidence rules.
- Produce each chapter from a chapter task card that names the reader promise, chapter job, thesis movement, required source refs, figure/table obligations, target extent, active memory refs, style constraints, and typed blockers.
- Treat adjacent-chapter repetition as a chapter-task defect. Preserve only the repetition that adds new evidence, mechanism, case sequence, model application, or transition; move repeated general argument back into the correct chapter task card or style note.
- After each chapter QC pass, back-propagate accepted repairs into the chapter task card, style asset bundle, glossary, evidence map, or semantic memory refs. Repeated defects must update the style engine or durable book memory, not only the local chapter text.
- After the assembled full draft reaches chapter budget, case/table, and required figure-asset gates, run the meta-review-loop skill before owner handoff or publication-proof claims. The drafting executor must not self-approve the full manuscript.
- Treat meta-review findings as a bounded review-repair loop. Required findings become a repair plan and chapter/source/style/QC updates; optional taste preferences are recorded but do not block; owner/source evidence gaps become typed blockers or owner-decision items.
- Cap default meta-review at three independent rounds. After every repair round, regenerate assembly metrics, hygiene, and the cumulative owner-review PDF before requesting the next review. If round three still has required manuscript repairs, stop with `meta_review_iteration_limit_reached` rather than looping forever.
- Preserve transparent prompt bundle refs for storyline shaping, chapter drafting, review, repair, style checks, publication proofing, and export handoff so owner review can inspect the instructions and source slices that drove each major pass.
- Keep reader-style contract refs in the production pipeline, chapter QC, and style consistency report so later passes can check naturalness against the intended readers.
- When a concrete voice regression is found, add it to the project hygiene scan or equivalent style gate. Known Red Bird outside-observer phrases such as `公开可观察`, `公开资料显示`, `教育实验观察窗口`, `观察它如何强调`, and `外部旁观` must be blocked from active manuscript and workflow refs when the reader-style contract calls for practice-involved voice.
- After each chapter reaches `chapter_draft_ready` or a narrower text-ready checkpoint, generate or refresh an owner-review PDF that contains the contiguous reviewable reader sequence, excludes below-target required units, and is explicitly labeled as a review artifact rather than a final export. If required figures are missing local bitmap assets, keep the chapter as text-reviewable but not fully ready.
- Keep owner-review PDFs contiguous from the start of the book. Do not skip an unfinished preface, introduction, or earlier chapter just because a later chapter has reached `chapter_draft_ready`; instead surface a review-continuity blocker and finish the earliest blocking unit first.
- Use the BookForge PDF export helper or an equivalent BookForge-owned backend adapter for PDF generation. Backends should be real publication/typesetting systems, for example Pandoc with XeLaTeX/LuaLaTeX, Quarto book rendering, or Typst. Do not hand-roll book PDF layout with ad-hoc raster text drawing or project-local renderer code as the normal path.
- For pre-numbered Chinese manuscripts, disable backend automatic section/caption numbering in review PDFs, or render figures in a way that does not create duplicate labels. The manuscript's own headings such as `第五章` and captions such as `图 5-1` must not become doubled by Pandoc or another backend.
- Establish a publication design profile before publication-grade proof output. Review PDFs are reading checkpoints; publication proofs require explicit page geometry, typography hierarchy, captions, case boxes, table style, image placement, headers/footers, numbering, visual rhythm, and rendered-page inspection.
- Keep voice, terminology, chapter structure, and transition style consistent.
- Keep section rhythm natural for the declared readers: scene or question -> explanation -> consequence -> transition. Avoid repeated outline-like sequences of definitions, numbered categories, and summary labels unless that form is genuinely the clearest reader-facing move.
- Use reference-derived book-prose checks when available: table of contents should show full-book movement; chapter openings should create reader tension before theory; dense body pages should vary paragraph rhythm; case sections should carry lived sequence and reflective judgment; figures and tables should be introduced and interpreted by prose; chapter endings should hand off to the next chapter.
- Use reference-derived PDF checks when available: front matter should look like a book/proof, chapter labels should use natural Chinese punctuation, and the table of contents should show the argument path rather than review-package internals. Render case boxes, tables, and figures as callouts/captions or other non-TOC elements unless the owner explicitly wants them in the TOC.
- Run an explicit internal-language and AI-flavor scan on active manuscript refs before chapter and whole-book handoff. Block visible production terms, stale status claims, outside-observer case phrases that violate the stance contract, and repeated formulaic negation unless each occurrence is intentionally justified in the style QC report.
- Do not make a fixed "reader-facing rewrite" pass the normal way to achieve natural prose. If such a pass is needed because the first draft reads like an instruction manual, classify it as a drafting-gate failure, update the chapter-production pattern, and only then continue production.
- Plan illustrations and tables as meaning-bearing book elements, with source and rights boundaries attached.
- Generate final book-bound bitmap illustrations through the Codex `imagegen` skill when image assets are required. Prefer the BookForge native imagegen asset helper or an equivalent BookForge-owned backend adapter: launch a child Codex executor with native imagegen enabled, copy the final bitmap into the book project, and record a helper receipt. Use helper-managed receipt/asset-manifest synchronization when available so `asset_ready`, project-local path, hash, dimensions, provenance, and blocker state do not depend on hand-edited manifest fields. Keep the prompt, project-local generated asset path, source/rights boundary, and review criteria with the illustration ref and figure asset manifest. Do not substitute SVG, local script diagrams, decorative placeholders, chat-preview-only images, or unlocatable generated images for final manuscript figures unless the owner explicitly asks for deterministic vector output.
- Do not make direct OpenAI Base URL/API-key provider calls the default figure route. API fallback is an explicit operator/owner route for large batches or unavailable built-in imagegen.
- Run a wording pass that favors direct claims, precise verbs, concrete nouns, and affirmative editorial movement.
- After assembling manuscript, review PDF, metrics, hygiene report, or handoff refs, refresh downstream receipts in order and compare their timestamps or source digests. Do not claim readiness from reports produced before the latest chapter, figure manifest, or PDF source changes.
- Run layout and typography checks before handoff.
- Return typed blockers for missing manuscript, source, reader-style, image, table, layout, or owner gate refs.

This skill does not publish, export, or approve final book quality without owner receipt.
