# Reader-Facing Draft Skill

Use this skill inside `book-materialization` when drafting or expanding chapter Markdown. It turns chapter packages into reader-facing prose on the first drafting pass instead of producing a planning memo that later needs translation into book prose.

Working policy:

- Draft from the reader-style contract, chapter brief, storyline map, and evidence map at the same time. The first visible chapter text should already read like a book chapter for the declared readers.
- Before writing visible prose, create a reader-entry plan inside the chapter brief or drafting notes: opening scene/question, reader tension, concrete example/case, main claim, section movement, table/figure role, and closing transition. Use that plan to draft the chapter body; do not expose the plan itself as manuscript prose.
- For every major section, name the reader-facing job before drafting: what problem the reader sees, what explanation changes their understanding, what consequence follows for curriculum/evaluation/project design, and how the section hands off to the next section. This prevents the first draft from becoming a list of framework elements.
- Keep production metadata out of manuscript prose. Chapter task, core question, chapter thesis, target budget, source refs, asset status, QC notes, and blockers belong in chapter briefs, manifests, QC reports, comments, or handoff refs; they must not appear as opening paragraphs or headings in the reader-facing chapter body.
- Open chapters with a concrete scene, reader problem, tension, or practical question before abstracting into concepts. Define terms after the reader can see why the term is needed.
- Shape paragraph movement as scene or question -> explanation -> consequence -> transition. A first draft should contain enough connective prose for a reader to feel the argument moving; do not leave bullet-like concept inventory that requires a later "reader rewrite" to become prose.
- Use headings as reader navigation, not planning labels. Headings should name what the reader is about to understand, not the internal job the section performs.
- Keep examples and cases inside the prose flow. Tables, figures, and case boxes should support an argument the prose has already prepared, not appear as manifest placeholders.
- Treat "reader-facing revision" as an exception or review response, not the normal path. If a chapter needs that pass because the first draft reads like a memo, record it as a drafting-gate failure and update the chapter-production pattern before continuing with more chapters.
- If a required figure is missing, keep a human-readable caption or figure note only when it helps review; do not expose implementation refs such as manifest filenames, asset status files, or prompt paths inside the main reader text.
- Treat reader-facing drafting as a first-draft quality target, not a separate late polish stage. A later reader-facing revision pass may still be used when review finds drift, but the workflow must not rely on that pass to remove predictable production scaffolding.

This skill does not bypass chapter budget gates, source grounding, figure asset blockers, or owner review. It changes the expected shape of the initial chapter prose.
