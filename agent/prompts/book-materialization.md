# Book Materialization Prompt

Goal: turn the approved storyline map into a book package with chapter drafts, illustration specs, table specs, consistency checks, wording revision, layout checks, and owner handoff refs.

Produce these refs:

- chapter draft bundle with chapter-level purpose and source refs
- chapter-sharded Markdown source refs for each chapter/part, plus a generated assembly ref for the whole manuscript
- target extent contract with page/word/count requirements inherited from the owner brief, source plan, publisher brief, or storyline map
- reader-style contract inherited from `storyline-architecture`, with audience, reading situation, and natural-expression rules used by chapter drafting and style QC
- reference-draft absorption report when the owner supplies a stronger prior version, comparable draft, sample chapter, or edited reference; the report must record transferable strengths, non-transferable patterns, adopted rules, and updates to style engine, chapter task cards, QC gates, and publication design
- author/source stance map inherited from `storyline-architecture`, including practice-involved cases and their allowed voice, evidence boundary, unsupported-outcome markers, and forbidden observer/public-source phrasing
- reader-facing draft policy that keeps internal production metadata out of manuscript prose and requires first-pass chapter text to read like book prose for the declared readers
- production pipeline contract that names the active chapter packages, chapter briefs, chapter drafts, chapter QC refs, figure asset manifest, table plan, whole-book review, export refs, and any obsolete/retired drafts
- chapter function contract inherited from `storyline-architecture`, with a chapter-by-chapter check that each chapter advances its assigned job without re-arguing adjacent chapters
- early concept map application report showing where recurring terms are introduced, foreshadowed, defined, and reused without forcing readers to infer definitions too late
- whole-book core model application report showing how the selected two or three core models are introduced, applied in later chapters/cases, and recovered in the final synthesis
- case evidence ladder report showing whether each major example is a constructed scene, typical scenario, documented process case, authorized material/interview, or outcome/impact claim, plus typed gaps for missing true cases
- reviewer/owner critique absorption report when critique exists, mapping each accepted suggestion to storyline refs, chapter task cards, style rules, case evidence gaps, figure/table plans, or manuscript repairs
- fast-track revision audit when owner/reviewer suggestions are small, local, and evidence-bounded; classify each suggestion, apply accepted manuscript repairs, defer owner/source-material decisions, and refresh assembly, metrics, hygiene, review PDF, and scan evidence without starting a new meta-review round
- chapter context compiler package for each active production chapter, including intent, selected refs, Standards/Novel/Manuscripts rule stack, protected vs compressible context, context trace, budget/missing extent, evidence boundaries, and one concrete next action
- source-claim integrity package with claim ledger, source locator, evidence boundary, unsupported-gap list, truth deltas after claim repairs, and anti-leakage notes for owner/private material
- style calibration profile when owner samples, stronger references, comparable works, or recurring style defects are available; include adoption/rejection rules, fatigue scan evidence, accepted exceptions, and back-propagation refs to style engine, terminology, task cards, or semantic memory
- independent meta-review loop refs after the assembled full draft reaches chapter and asset gates: `meta-review/round-N.md`, `meta-review/round-N-repair-plan.md`, revision notes, and stop/pass/blocker decision, capped at three rounds unless the owner explicitly changes the policy
- revision entrypoint decision refs after meta-review, serious owner/reviewer critique, or complete-version comparison: `meta-review/round-N-entrypoint-decision.md`, `revision-routing/decision-N.md`, or equivalent, selecting the topmost repair level before prose edits
- chapter production budget with target pages/chars per chapter and a status gate that distinguishes `not_started`, `outline_only`, `seed_in_progress`, `draft_in_progress`, `chapter_draft_ready`, and `blocked`
- active production queue with each required unit's target minimum, current measured extent, missing extent, next action, review-PDF eligibility, and typed blocker state
- book-memory contract with working, episodic, and semantic memory refs mapped to the active nonfiction manuscript
- chapter task cards for each active chapter, naming reader promise, chapter job, thesis movement, source refs, target extent, figure/table obligations, active memory refs, style constraints, QC state, and typed blockers
- chapter repair back-propagation report showing which accepted QC findings updated the chapter draft, task card, style engine, glossary, evidence map, or semantic memory refs
- transparent prompt bundle refs for chapter drafting, review, repair, style pass, figure/table planning, and export handoff
- illustration plan with image purpose, placement, `imagegen` prompt/spec, generated bitmap asset path when produced, rights/source boundary, and review criteria
- figure asset manifest with one record per required figure, including required/optional status, prompt ref, generated preview ref if any, project-local bitmap path, file existence/inspection status, and blocker kind when missing
- table plan with data source, claim supported, column logic, and formatting rules
- completed-chapters review PDF ref, generated through a publication/typesetting backend after any chapter reaches `chapter_draft_ready`, containing only completed chapters and clearly labeled as an owner review artifact rather than a final export
- PDF export receipt that states `artifact_role` as `review_pdf`, `publication_proof`, or `final_export`; names the real typesetting backend; records resource paths for relative figures; records Markdown image-ref resolution and figure-asset-manifest readiness; and records artifact-gate blockers separately from PDF compile status
- publication-proof output refs when `artifact_role=publication_proof`: publication design token ref, template/component inventory ref, font actual-load/readback ref, rendered-page QA checklist ref, pre-ship proof review ref, front matter/TOC cleanliness ref, page rhythm/density/orphan check ref, and material/asset coverage ref
- style consistency report covering voice, terminology, pacing, transitions, and repeated phrases
- AI-flavor revision report that replaces vague hedging, contrast formulas, inflated phrasing, empty summaries, and internal workflow language with direct reader-facing prose; include concrete scan evidence or accepted exceptions for high-risk patterns
- publication design profile that distinguishes review PDF, publication proof PDF, and final export
- layout and typography QC report
- owner handoff packet with remaining decisions and blockers
- project hygiene report that scans active manuscript/workflow refs for stale status claims, forbidden case-stance phrases, and retired full-text drafts that remain searchable inside the active workspace

Maintain the storyline contract, reader-style contract, and declared target extent. Keep chapter prose concrete, consistent, evidence-grounded, and publication-aware. Use affirmative phrasing as the default editorial move.

If the owner provides a stronger reference version or asks which version is better, run reference absorption before new chapter drafting or major rewrite. Compare the reference against the active manuscript across title/front matter, table of contents, chapter openings, dense body prose, case treatment, figure/table pages, transitions, chapter endings, and total-book completion. Extract why the reference reads better as transferable rules, then write those rules into style-engine assets, chapter task cards, reader-entry plans, QC reports, and publication-design refs. Do not copy reference prose or adopt secondary-reader explanations that conflict with the primary-reader contract. If the current manuscript remains outline-like, memo-like, visibly review-scaffolded, or incomplete after the reference comparison, treat that as a drafting-pattern defect and repair the production pattern before continuing.

When the reference is a stronger full-book version or review PDF, do a complete-version absorption pass before manuscript repair. Decide whether the reference is better because of reader-entry sequence, book-like transitions, localized scope language, fuller chapter decomposition, richer operation/process cases, methodized practice chapters, conclusion action path, or cleaner PDF/TOC hierarchy. Also record what the active manuscript should keep, such as sharper concept density, concise model names, stronger evidence boundaries, or cleaner argument focus. The default editorial rule is: preserve cases and transitions that help readers follow the argument, compress repeated central judgments that do not add evidence or mechanism, and never inflate length only to resemble the reference.

If the owner says the active manuscript has moved toward the stronger reference but still reads like a compressed review edition, classify the gap before editing. For a formal publication manuscript, the repair target is not only shorter headings or sharper claims; it is a book-like reading path with enough narrative entry, stakeholder tension, explanatory transitions, short cases, figure/table interpretation, and chapter handoffs for the declared primary readers to think along. Keep the active manuscript's useful precision, but restore reader walking space where the draft jumps too quickly from scene to concept or from concept to another concept. Treat high judgment density as a craft defect: add micro-cases, process beats, classroom/workplace/project examples, and connective explanation at the concept boundary; do not pad with repeated thesis claims or generic summaries.

When repeated complete-version feedback says the active manuscript is now the best `completed` or concise edition but still not the best formal-publication base, make an artifact-route decision before further edits. Record whether the current artifact is a `concise_review_edition`, `internal_trial_reading_edition`, or `formal_publication_manuscript`. For a concise route, apply only bounded fast-track repairs, keep the review/e-book label visible, and do not claim formal publication readiness. For a formal-publication route, use the fuller reference or a formal expansion queue as the base plan; do not keep applying small local polish to a compressed manuscript while calling the publication-level critique absorbed.

When a comparison raises figure-style uncertainty, update the publication design profile before regenerating assets. Choose a whole-book figure stance, such as light methodology sketchnote or formal framework diagram, record why it fits the reader-style contract and artifact target, and keep prompts, captions, and proof inspection consistent with that stance. Do not mix informal handwritten figures and formal institutional diagrams without an explicit design reason.

When absorbing a stronger local revision, translate the improvement into production rules instead of only applying the edit. Capture reader-action openings, relationship sentences, chapter-scope narrowing, process-chain movement, case-in-argument placement, and figure/table interpretation rules in the affected chapter task cards and style engine. Also capture the side effects: whether the reference shortened the argument, removed "why" explanation, weakened primary-reader density, or introduced merge fragments. A local slice can improve the whole workflow, but it cannot be labeled as a completed whole-book version or used to skip remaining chapter, figure, evidence, or publication gates.

When an owner or reviewer critique describes the manuscript as coherent but below high-quality nonfiction level, route it before editing prose. For each critique item, decide the topmost repair level: `artifact_target_repair`, `storyline_architecture_repair`, `outline_sequence_repair`, `chapter_function_repair`, `evidence_model_repair`, `publication_design_repair`, `local_prose_repair`, or `owner_source_blocker_only`. Use a reverse outline or equivalent structure extraction when the finding may concern whole-book organization. Update the selected durable refs first, then revise chapter Markdown only after the repair entrypoint is clear. Do not only produce a polished version while leaving the workflow unable to prevent the same defect in the next book.

When an owner or reviewer critique is clearly local and low risk, use fast-track revision instead of a full meta-review cycle only after the entrypoint decision is local or explicitly bounded. Eligible repairs include local wording rhythm, model short-name memory aids, chapter-boundary notes, small action checklists, and evidence-label clarity. Ineligible repairs include new factual case claims, Red Bird outcome claims, broad chapter-function changes, reader-contract changes, target-extent changes, publication-proof issues, and owner title/subtitle decisions. Every fast-track pass must leave an audit ref that records accepted edits, deferred items, touched refs, evidence boundaries, refreshed artifacts, and validation commands.

Before drafting, repairing, or resuming a chapter, compile a chapter context pack. Use the external writing-agent lesson as a disciplined input shape, not as a new runtime: first plan and compile the active intent/source/rule refs, then compose the bounded chapter task, then write or repair the prose. The pack should separate global Standards, durable Novel truth, and Manuscripts-local chapter state; name protected context that must not be compressed away; name compressible context that can be summarized; and leave a trace for why each source, rule, memory, style, and evidence ref was included or excluded. If the pack cannot identify the active task card, reader-style contract, source boundary, target extent, or practice-involved case stance, return a typed blocker. If the missing context only blocks a wider readiness claim, keep moving with a narrower text action and record the blocker against the wider claim.

Run source-claim integrity before strengthening claims, after major rewrites, and at chapter-package/assembly boundaries. Every material claim should be inspectable through a claim ledger and source locator, including table, figure, caption, callout, and case-box claims. Match language to evidence class: constructed scenes and typical scenarios orient readers, documented process material supports process claims, authorized material supports attributed details within its permission, and outcome/impact language requires outcome/impact evidence. If an unsupported claim is not central, remove or bracket it, record the gap, and continue. If it is central, return a typed blocker such as `source_locator_missing`, `missing_outcome_evidence`, `owner_authorization_needed`, or `practice_stance_overclaim`.

Calibrate style from owner-inspectable evidence when samples, reference drafts, house style, prior chapters, or recurring QC findings exist. Build a reusable style profile before importing rules: classify inputs, extract transferable rules, reject non-transferable reference patterns, run fatigue scans with counts or locations, and record accepted exceptions. Style calibration should make first drafts more natural for the primary readers without lowering argument density, hiding source gaps, copying reference prose, or claiming final editorial acceptance.

After all required chapters and required figure assets reach their gates, run an independent meta-review before claiming draft handoff quality. Use a context-isolated reviewer as far as the execution environment allows: give the reviewer the assembled manuscript and minimal quality contracts, not the drafting conversation or self-justification. The meta-review must judge whether the book's viewpoint is clear, logic is natural, expression is fluent, chapter repetition is controlled, evidence/cases are adequate, practice-involved cases use the right voice, and AI-flavor/internal workflow language is absent. Before repair, run the revision-entrypoint-router: extract or update a reverse outline, classify each finding by repair layer, select the topmost entrypoint, and record route-back/blocker/freshness obligations. Convert routed required findings into a repair plan, revise the selected durable refs and then chapter Markdown, regenerate assembly/metrics/hygiene/review PDF, and review again. Stop early on `pass` or when remaining items are optional preferences, owner/source evidence gaps, publication-proof issues, or final-export acceptance. If three review-repair rounds still leave required manuscript repairs, return a typed `meta_review_iteration_limit_reached` blocker instead of continuing indefinitely.

During chapter drafting and revision, enforce chapter function compression. Each chapter may echo the central premise only briefly, then must spend its body on its assigned movement. Remove or move repeated material into the correct chapter's task card when adjacent chapters keep restating the same "why this matters" claim. If the book needs repetition for public readability, make the repetition carry new evidence, a new case, an operational mechanism, or a model application.

Apply the concept map early in the reader sequence. When a recurring concept is first used before its full definition chapter, provide a compact orientation or preview in the preface/opening chapter. Later chapters can deepen the definition, but the reader should not have to infer the book's key terms for several chapters.

Use the whole-book core models as recurring argument tools. A figure or table that only summarizes local information is useful but not enough for a modelization gate. At least the selected core models should be named, interpreted in prose, applied to a later case or chapter, and recovered in the final synthesis.

Use the case evidence ladder to set claim strength. A typicalized classroom scene may illustrate a tension; a documented project process may support operational design claims; authorized participant material may support lived-detail claims; outcome data may support impact claims. If the needed level is absent, keep `待补真实案例` or a typed evidence gap in the appropriate case note instead of smoothing it into a confident story.

Do not begin body drafting or chapter expansion when the reader-style contract is missing, generic, or unresolved. If audience and natural expression were inferred rather than owner-specified, keep the inference and owner-review status visible in the handoff and style reports.

Do not draft major cases until their author/source stance is known. If the author team designed, executed, or participated in the case, use the approved practice-involved stance in the visible chapter prose. Explain design anticipation, pedagogical or operational choices, execution mechanisms, and reflective limits directly. Do not write the same case as neutral public-source observation just because some public sources are also used. Evidence boundaries still apply: missing authorization, interviews, user feedback, learning outcomes, metrics, or full process materials must remain explicit typed gaps rather than being smoothed over by confident narration.

Make the first chapter draft reader-facing by default. Before writing visible prose for each production chapter, create a reader-entry plan inside the chapter brief or drafting notes: opening scene/question, reader tension, concrete example/case, main claim, section movement, table/figure role, and closing transition. Chapter task, core question, chapter thesis, target extent, source refs, figure asset states, QC notes, blockers, and this reader-entry plan belong in chapter briefs, manifests, QC reports, comments, and owner handoff refs; they must not be emitted as visible chapter openings, section headings, or explanatory paragraphs in the manuscript body. Start chapters from a concrete scene, reader problem, tension, or practical question before moving into concepts. If a later reader-facing revision pass is useful, it is allowed, but Book Forge must not depend on a late polish pass to remove predictable planning scaffolding. If a chapter needs a routine reader-facing rewrite because the first draft reads like a memo, treat that as a drafting-gate failure and fix the chapter-production pattern before continuing with more chapters.

Use a Markdown-first writing surface. Write and revise prose in per-chapter Markdown files or equivalent author-facing Markdown refs. Use generator scripts only as thin assemblers/checkers/exporters that read manuscript refs and produce metrics, reports, merged `book.md`, DOCX/PDF, or receipts. Do not embed book prose in Python/TypeScript/shell/JSON string literals as the manuscript source of truth.

Use chapter task cards as the chapter-runtime surface. They are owner-inspectable refs for intent, source slices, active memory, style constraints, figure/table obligations, QC state, and blockers. They are not a private scheduler, queue, attempt ledger, session store, or runtime truth source.

Run chapter repair as a feedback loop. First repair the chapter package; then update durable style, glossary, evidence, and semantic-memory refs only when the finding is reusable. Keep rejected repairs visible with reason and owner decision need.

For book-length work, shard drafting by chapter or part: chapter brief -> chapter Markdown draft -> chapter-level QC -> merged manuscript -> whole-book pass. Do not make a single long `book.md` the only creative working file for a multi-chapter book unless the owner explicitly requests a one-file workflow.

Before drafting body prose for a book-length target, convert the owner/source target extent into a chapter production budget and an active production queue. A "coverage pass" that puts short seed text into every chapter is not book materialization unless each chapter is intentionally labeled seed/in-progress and the next production chapter is named with its remaining character/page budget.

Make the chapter budget a production gate, not a later audit. Each chapter drafting prompt must include the chapter's minimum extent, current measured extent, missing extent, required section development, case/example obligations, and figure/table interpretation obligations. A chapter below its inherited minimum extent is not drafted/done; it is `seed_in_progress` or `draft_in_progress`. Do not mark all chapters complete and then discover the manuscript is short. Generate `book.preview.md` for below-target whole-book previews, and reserve final `book.md` for an all-chapters-ready assembly unless the owner explicitly requests a named preview export. If a style pass or cleanup pass reduces a chapter below budget, repair it with substantive argument, cases, examples, transitions, or figure/table interpretation grounded in the chapter brief, not with padding or repeated summaries.

After every chapter reaches `chapter_draft_ready` or a narrower text-ready checkpoint, compile a review PDF containing the contiguous reader sequence from the beginning of the book: front matter/opening matter plus chapters in order, stopping at the first below-target required unit. Do not silently skip an unfinished preface, introduction, or earlier chapter to show later ready chapters as if the book were continuous. If a later chapter is ready but an earlier required unit is not, return a typed review-continuity blocker and make the earlier unit the next production target. Name and label this artifact as an owner-review PDF, not a final book export, so the owner can review cumulative text and request corrections before the next chapters are produced. If a text-ready chapter still lacks required project-local imagegen assets, include it only as a text-review checkpoint and keep full chapter readiness blocked.

Run assembly, metrics, hygiene, review-PDF, and owner-handoff updates in freshness order after chapter/source/figure changes. The readiness evidence must be newer than, or digest-linked to, the chapter Markdown, figure asset manifest, PDF source, and style refs it inspects. A stale metrics table, old hygiene report, or previous review PDF receipt is a blocker, not evidence.

Treat PDF generation as a first-class Book Forge export capability. Use the Book Forge PDF export helper or an equivalent Book Forge-owned backend adapter to call a real publication/typesetting backend such as Pandoc with XeLaTeX/LuaLaTeX, Quarto book rendering, or Typst. Do not implement publication PDF by hand-drawing text into images, ad-hoc canvas/Pillow page painting, or other bespoke renderer code except as a temporary diagnostic fallback with a typed blocker.

For PDF export, distinguish the target explicitly. A `review_pdf` is a readable owner/editor review artifact and cannot be upgraded by wording into a publication proof. A `publication_proof` requires a publication design profile, a real typesetting backend such as Pandoc with XeLaTeX/LuaLaTeX, Quarto, Typst, or Paged.js, resource paths that make relative figures resolve, rendered page refs, Markdown image-ref checks, figure-asset-manifest readiness checks, and rendered-page inspection of captions, callouts, tables, figures, hierarchy, running heads/feet, page numbers, overflow, and visual rhythm. Use the Book Forge bundled `bookforge-zh-publication-proof` profile or an owner-approved equivalent when the owner asks for publication-grade electronic-book quality; do not accept unstyled Pandoc defaults as a publication proof. The PDF helper may generate a machine-baseline rendered-page inspection for nonblank pages and asset plumbing, but book-specific visual judgment still requires human inspection before owner handoff. A `final_export` requires the publication proof evidence plus owner/export acceptance receipts. If the backend, design profile, rendered-page inspection, element gates, resource paths, local figure assets, or owner receipts are missing, return a typed export blocker instead of claiming export readiness.

Before producing a publication-grade proof, establish a publication design profile. It must define page size, geometry, body font, heading scale, line length, caption style, figure placement, table style, case-box/callout style, headers/footers, page numbering, visual rhythm, and rendered-page inspection plan. For publication proof, express those choices as concrete design tokens and pair them with a template/component inventory covering front matter, TOC, chapter openers, dense body pages, figures, tables, callouts/case boxes, captions, and closing matter. Record font actual-load/readback when the backend or PDF inspection path can report it; if fallback fonts are used, rerender and reinspect page count, density, glyph coverage, and orphan risk before proof claims. A readable review PDF is useful for owner review but is not enough to claim publication quality. Publication-proof inspection should sample the title/front matter, TOC, chapter opening, dense body page, figure/table page, callout page, and final page for nonblank content, glyph clipping, font fallback, caption proximity, image rendering, page headers/footers, page numbers, density/orphan risk, and monotony. Run a pre-ship proof review against the tokens, component inventory, front matter/TOC policy, material/asset coverage, and rendered-page checklist before handing the proof to the owner.

Keep proof gates scoped to proof claims. Missing publication design tokens, component inventory, font readback, rendered-page QA, pre-ship proof review, or owner/export receipt must block `publication_proof` and `final_export` claims, but should not block chapter drafting, source-claim integrity, style calibration, chapter QC, or ordinary `review_pdf` output when those actions can proceed with narrower honest evidence.

When restarting after an invalid compact/sample draft, retire that draft into an archive/tombstone ref and rebuild from the chapter packages. Do not treat the compact draft as the expansion seed unless the owner explicitly accepts that route.

Retired draft handling should remove old manuscript prose from searchable active-workspace archive refs. Keep a tombstone with reason, date, and current source-of-truth refs. Do not leave old full chapter text in `archive/obsolete-*` when it contains a rejected voice, wrong workflow, wrong extent, or other text that can be mistaken for the current book.

Do not shorten, compact, summarize, or convert a book-length target into a sample draft unless the owner explicitly changes the target extent. If the requested extent cannot be completed in the current pass, materialize the stable plan and partial manuscript, then return a typed extent blocker with the missing pages/words/chapters.

After updating chapters, review PDFs, metrics, or owner handoff refs, run a hygiene scan equivalent to `bookforge_project_hygiene.py` when available. Treat stale metrics in README/handoff refs and known forbidden case-stance phrases as blockers before handing the workspace back to the owner.

Before chapter or whole-book handoff, run an internal-language and AI-flavor scan over active manuscript refs. Block production terms and stale progress commentary such as `当前版本`, `最终书稿`, `source of truth`, `QC`, `blocker`, `manifest`, `后台`, PDF/export readiness notes, and outside-observer case phrases that violate the stance contract. Prefer affirmative, concrete sentences over formulaic negation pairs such as `不是...而是...`, `不只是...更是...`, `并非...而是...`, and `不仅...还...` unless the contrast carries real conceptual work. Record counts, locations, and accepted exceptions in the style or AI-flavor report.

After reference absorption, chapter merging, or whole-book assembly, run a sentence-integrity scan before export. Check for truncated lines, duplicated headings, doubled chapter/figure labels, half-generated fragments, empty transition paragraphs, and abrupt loss of explanation caused by over-compression. Repair these in the chapter Markdown source first, then regenerate the assembly and review PDF.

For final manuscript figures, use raster assets generated through the Codex `imagegen` skill by default. Prefer the Book Forge native imagegen asset helper or an equivalent Book Forge-owned backend adapter so final artwork is generated through a Codex executor with native imagegen enabled, copied into the project, recorded in a receipt, and synchronized back into the figure asset manifest by `figure_id`. Do not use SVG, script-generated diagrams, or placeholder art as final figures unless the owner explicitly requests deterministic vector output. If image generation cannot be completed, or if the generated file path is not exposed so the asset cannot be copied into the project, return a typed image-asset blocker instead of silently downgrading the figure. Do not make direct OpenAI Base URL/API-key calls the default Book Forge figure route; direct API fallback is an explicit operator/owner route for large batches or unavailable built-in imagegen.
